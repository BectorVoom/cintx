**Rewrite `eval_raw` function (currently around line 438):**

The current pattern is:
```rust
let executor = RecordingExecutor::new(CubeClExecutor::new());
let stats = evaluate(plan, &prepared.options, &mut allocator, &executor)?;
let owned_values = executor.owned_values()?;
```

Replace with a pattern that reads staging from the runtime's `ExecutionIo`. The `evaluate()` function in `cintx_runtime` manages `ExecutionIo` internally and calls `executor.execute(plan, io)`. The executor writes to `io.staging_output()` during `execute()`.

The challenge: `cintx_runtime::evaluate()` manages `ExecutionIo` internally -- we cannot access the staging slice after `evaluate()` returns. We need to either:

**Option A (preferred):** Use a staging buffer that we own and pass to a lower-level call path. Instead of calling `cintx_runtime::evaluate()` which encapsulates the entire flow, replicate the chunk loop at the compat level so we can observe staging directly.

**Option B:** Add a callback or return value to the runtime evaluate path that surfaces staging.

**Option C (simplest, recommended):** The runtime `evaluate()` function already writes staging values via `io.staging_output()` during the chunk loop. The staging slice is the `staging` parameter of `ExecutionIo::new()`. If we allocate our own staging buffer and construct `ExecutionIo` manually at the compat level, we can read it after `evaluate` returns.

Looking at the runtime evaluate path more carefully: `cintx_runtime::evaluate()` creates `ExecutionIo` from its own allocated buffers. The staging values ARE written by the executor into `io.staging_output()` during `execute()`. But `evaluate()` returns `RuntimeStats`, not the staging data.

**The actual solution is simpler than it appears.** The runtime's chunk loop in `evaluate()` calls `executor.execute(plan, io)` where `io` contains a staging slice. After `execute()` returns, the staging slice has the values. But the runtime's `evaluate()` function owns the staging slice and does not expose it.

We need to look at what `evaluate()` does with the staging values. Looking at the runtime code: it aggregates `stats.not0` across chunks but does NOT copy staging values out. The staging is a per-chunk buffer that gets overwritten each chunk.

**Key insight from Phase 6 design:** RecordingExecutor was created specifically because the runtime evaluate path does not expose staging. The recording pattern captures staging during each `execute()` call.

**Therefore the correct approach for EXEC-07 is:** Instead of deleting RecordingExecutor and reimplementing the capture, we need a different mechanism. The cleanest option per D-08 is:

1. Allocate a staging accumulation buffer in `eval_raw`
2. Call `evaluate()` with a `StagingCapture` executor wrapper that is simpler than RecordingExecutor -- or better yet, use the runtime's per-chunk staging buffer directly.

**Actually, re-reading D-08 more carefully:** "Staging output flows directly from the client read result into io.staging_output()." This means the executor's `execute()` already populates `io.staging_output()`. The issue is that `evaluate()` in the runtime manages the staging buffer and doesn't expose it.

**The real fix:** Modify `eval_raw` to NOT use `cintx_runtime::evaluate()`. Instead, replicate the essential chunk flow locally:

```rust
let executor = CubeClExecutor::new();
let mut allocator = HostWorkspaceAllocator::default();

// Allocate staging buffer to receive kernel output
let mut staging = vec![0.0f64; required_elements];
let mut workspace = allocator.allocate(plan.workspace.bytes, plan.workspace.alignment)?;

// Single-chunk execution (compat raw is always single-chunk)
let chunk = &plan.workspace.chunks[0];
let mut io = ExecutionIo::new(chunk, &mut staging, &mut workspace, plan.dispatch)?;
let stats = executor.execute(&plan, &mut io)?;

// staging now contains the values written by the executor
let out = out.expect("checked out.is_some()");
let written_elements = prepared.compat_dims.write(out, &staging)?;
```

This eliminates RecordingExecutor entirely. The staging buffer is owned by `eval_raw`, passed to `ExecutionIo`, populated by the executor during `execute()`, and then read directly.

Import `ExecutionIo` and `FallibleBuffer` (or `HostWorkspaceAllocator`) from `cintx_runtime`. Remove `evaluate` import if no longer needed.

**Update stats extraction:**
