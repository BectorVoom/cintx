## Summary

Phase 8 implements four tightly related math primitives as `#[cube]` functions in `crates/cintx-cubecl/src/math/`. These primitives are consumed by all kernel families in Phase 9 and 10. All implementations must be validated against libcint C oracle calls before any kernel consumes them.

The math is well-understood from the vendored `libcint-master/src/` sources. The primary implementation challenge is translating C control flow into CubeCL-compatible patterns: no plain Rust functions callable from `#[cube]`, no `.exp()` method syntax, and if-expressions must use the statement-form (assign to `mut` then branch). CubeCL struct support requires `#[derive(CubeType)]` on structs used inside `#[cube]` functions — fallback to flat field passing exists if struct support is limited.

The libcint Boys function is `gamma_inc_like()` in `fmt.c` — not a table-based function at all. It uses an iterative power series for small `x` and upward recurrence from `F_0` (via `erf`) for large `x`. The turn-over is per-order, driven by the `TURNOVER_POINT[]` array (40 entries, orders 0–39). MMAX is not defined as a single constant in a header; the practical upper bound is order 39 (the last non-zero `TURNOVER_POINT` entry), which matches `ANG_MAX=15` for 2e integrals: `(4*ANG_MAX)/2 + 1 = 31` roots, so order 30 is sufficient for standard use. The research concludes: implement as the iterative algorithm matching `gamma_inc_like`, treating the `TURNOVER_POINT` array as the branch table.

**Primary recommendation:** Implement `boys.rs` as `gamma_inc_like` (iterative power series + erf upward recurrence), `rys.rs` as Clenshaw-evaluated Chebyshev polynomials with fallback to Wheeler, `pdata.rs` as a `#[cube]` struct, and `obara_saika.rs` with separate `vrr_step`/`hrr_step` functions. All must pass 1e-12 atol vs oracle before Phase 9 starts.

## libcint Algorithm Details

### Boys Function (`fmt.c`)

The Boys function `F_m(t)` is the incomplete gamma function integral:

```
F_m(t) = integral_0^1 u^{2m} exp(-t u^2) du
```

libcint implements this in `gamma_inc_like()` (not a table lookup). The algorithm:

**Branch 1 — t == 0:**
```
F_m(0) = 1.0 / (2m + 1)
```

**Branch 2 — t < TURNOVER_POINT[m] (power series):**
```
F_m(t) = exp(-t)/2 * sum_{k=0}^{inf} t^k / prod_{i=1}^{k} (b+i)
         where b = m + 0.5
         downward recurrence: F_{m-1} = (exp(-t) + t * F_m) / (m - 0.5)
```
Terminates when term `< DBL_EPSILON * 0.5 * exp(-t)`.

**Branch 3 — t >= TURNOVER_POINT[m] (erfc + upward recurrence):**
```
F_0(t) = sqrt(pi/4) / sqrt(t) * erf(sqrt(t))
F_m(t) = (1/(2t)) * ((2m-1) * F_{m-1}(t) - exp(-t))
```

**TURNOVER_POINT array** (40 entries, index 0..39):
```
[0, 0, 0.866, 1.295, 1.705, 2.106, 2.501, 2.892, 3.280, 3.666, 4.050,
 4.432, 4.814, 5.194, 5.574, 5.952, 6.330, 6.708, 7.085, 7.461, 7.837,
 8.213, 8.589, 8.964, 9.338, 9.713, 10.087, 10.461, 10.835, 11.209,
 11.582, 11.955, 12.328, 12.701, 13.074, 13.447, 13.819, 14.192, 14.564, 14.936]
```

**Key constants:**
- `SQRTPIE4 = 0.886226925452758...` (sqrt(pi/4))
- `SML_FLOAT64 = DBL_EPSILON * 0.5` (convergence tolerance)
- Maximum practical order: 39 (last non-zero `TURNOVER_POINT`)
- For standard 2e integrals with `ANG_MAX=15`: max order = `(4*15)/2+1 = 31`

**MMAX for phase:** Use 39 as the const upper bound (`MMAX = 39`). This covers all families including derivatives.

**Note on D-01 (table-based):** libcint's `fmt.c` does NOT use precomputed Taylor tables stored in arrays. It uses the iterative power series directly. D-01 says "replicate libcint's table-based approach" but the actual source is iterative. The correct approach per D-08 (match libcint byte-for-byte in reference) is to replicate the iterative algorithm. The `#[cube]` implementation can still embed the TURNOVER_POINT array as a compile-time const.

Source: `libcint-master/src/fmt.c`, lines 42–226. Confidence: HIGH.

### Rys Quadrature (`rys_roots.c`, `polyfits.c`)

`CINTrys_roots(nroots, x, u, w)` computes `nroots` quadrature roots and weights for argument `x`.

**Strategy dispatch (from `rys_roots.c`):**
- `x <= 3e-7` (SMALLX_LIMIT): linear interpolation from `POLY_SMALLX_R0/R1` and `POLY_SMALLX_W0/W1` tables in `roots_for_x0.dat` (binary)
- `x >= 35 + nroots*5`: large-x asymptotic from `POLY_LARGEX_RT/WW` tables
- `nroots = 1..5`: dedicated `rys_root1..5()` functions (polynomial fits)
- `nroots = 6,7`: Jacobi/Schmidt segment solve with breakpoint at x=11
- `nroots = 8`: Jacobi/Schmidt, breakpoint at x=11
- `nroots = 9`: Jacobi/Laguerre, breakpoint at x=10
- `nroots = 10,11`: Jacobi/Laguerre, breakpoint at x=18
- `nroots = 12`: Jacobi/Laguerre, breakpoint at x=22
- `nroots >= 13`: quad-precision Jacobi/Laguerre, breakpoint at x=50

**polyfits.c structure:**
- Chebyshev polynomial fits via Clenshaw recurrence (`_CINT_clenshaw_dc`, `_CINT_clenshaw_d1`)
- 14x14 cosine matrix (`COS_14_14`) for transform
- 14x14 matmul step (`_CINT_matmul_14_14`)
- `CINTsr_rys_polyfits()` for short-range Rys variants

**Wheeler algorithm (`rys_wheeler.c`):**
- `_CINTdiagonalize()` computes Gauss quadrature nodes via tridiagonal eigenproblem
- Uses Jacobi alpha coefficients for orthogonal polynomial recurrence
- The 40-entry `JACOBI_ALPHA` table is embedded in the source
- This is an iterative algorithm, not a table lookup

**Max Rys roots for phase:**
- For 2e integrals with ANG_MAX=4 (d-functions): `(4+4)/2+1 = 5` roots → covered by dedicated functions
- For full coverage up to l=15: up to 31 roots → needs Wheeler fallback
- Phase scope: d-function tests need 5 roots; full coverage needs up to ~12

Source: `libcint-master/src/rys_roots.c` lines 57-123; `rys_wheeler.c` lines 1-80. Confidence: HIGH.

### Gaussian Pair Data (pdata)

From `g1e.c` and `g2e.c`, the pair data structure consists of:
- `aij = ai + aj` (sum of exponents)
- `rij = (ai*Ri + aj*Rj) / aij` (weighted center)
- `rirj = Ri - Rj` (displacement)
- Pre-exponential factor `fac = exp(-ai*aj/aij * |Ri-Rj|^2) * norm_i * norm_j`
- `aij2 = 0.5 / aij` (used in recurrence)

For 2e integrals, two pairs (ij and kl) produce:
- `zeta_ab = (ai+aj)*(ak+al)/(ai+aj+ak+al)` (reduced exponent)
- `center_p = rij`, `center_q = rkl` (pair centers)
- `center_w = (zeta_p * rij + zeta_q * rkl) / (zeta_p + zeta_q)` (combined center)

Source: `libcint-master/src/g1e.c` lines 125-183; `g2e.c` lines 268-400. Confidence: HIGH.

### Obara-Saika Recurrence (`g1e.c`, `g2e.c`)

**VRR (Vertical Recurrence, g1e overlap):**
```
G[0, n+1] = rij_rx * G[0, n] + n * aij2 * G[0, n-1]
G[j, i]   = G[j-1, i+1] + rirj * G[j-1, i]  (HRR step)
```
where `rij_rx = rij - rx` (rx is the "base" center, either ri or rj).

**VRR (Nuclear attraction / 2e, g1e.c CINTg1e_nuc):**
```
For each Rys root u[n]:
  ru = tau^2 * u[n] / (1 + u[n])
  rt = aij2 - aij2 * ru
  r0 = rijrx + ru * crij_x   (Rys-weighted displacement)
  G[n, 0, 0] = 1
  G[n, i+1, 0] = i*rt * G[n, i-1, 0] + r0 * G[n, i, 0]
```

**VRR (2e, `g2e.c CINTg0_2e_2d`):**
```
gx[irys, 0, n+1] = c00x[irys] * gx[irys, 0, n] + n*b10[irys] * gx[irys, 0, n-1]
gx[irys, m+1, 0] = c0px[irys] * gx[irys, m, 0] + m*b01[irys] * gx[irys, m-1, 0]
gx[irys, m+1, n+1] = gx[irys, m, n] + b00[irys]*gx[irys, m, n] 
                    + n*b10[irys]*gx[irys, m+1, n-1] + m*b01[irys]*gx[irys, m-1, n+1]
```

where `c00x`, `c0px`, `b10`, `b01`, `b00` are Rys-root-dependent recurrence coefficients in `Rys2eT` struct.

**HRR (Horizontal Recurrence):**
```
G[j, i] = G[j-1, i+1] + rirj * G[j-1, i]
```
Applied after VRR to build higher-j integrals from the i-ladder.

**`hrr_step` and `vrr_step` decomposition (D-14):**
- `vrr_step(g: &mut Array<f64>, c00: f64, b10: f64, n_max: u32, stride: u32)` — builds the n-ladder for one Rys root and one Cartesian dimension
- `hrr_step(g: &mut Array<f64>, rirj: f64, di: u32, dj: u32, lj: u32, nmax: u32, nroots: u32)` — applies HRR for one dimension

Source: `libcint-master/src/g1e.c` lines 125-350; `g2e.c` lines 268-450. Confidence: HIGH.

### Pattern 2: CubeCL `#[cube]` Constraint Patterns

All `#[cube]` function code must follow these rules (from project `docs/manual/Cubecl/cubecl_error_solution_guide/`):

**Rule 1 — No plain Rust function calls:**
Every helper called from `#[cube]` must itself be `#[cube]`:
```rust
// WRONG — causes E0433
fn double_factorial(n: u32) -> f64 { ... }

#[cube]
fn boys_asymptotic(n: u32, x: f64) -> f64 {
    let c = double_factorial(n);  // E0433
}

// CORRECT
#[cube]
fn double_factorial(n: u32) -> f64 { ... }

#[cube]
fn boys_asymptotic(n: u32, x: f64) -> f64 {
    let c = double_factorial(n);  // OK
}
```

**Rule 2 — No `.method()` for math operations:**
```rust
// WRONG
let e = (-t).exp();

// CORRECT
let e = f64::exp(-t);
let s = f64::sqrt(t);
let ef = f64::erf(t);  // check if erf is in cubecl::prelude
```

**Rule 3 — No `if` expressions as values:**
```rust
// WRONG — E0308
let branch_val = if t < turnover { power_series(t, m) } else { erfc_recurrence(t, m) };

// CORRECT
let mut branch_val: f64 = 0.0;
if t < turnover {
    branch_val = power_series(t, m);
} else {
    branch_val = erfc_recurrence(t, m);
}
```

**Rule 4 — Supported types only:**
Use `f64`, `f32`, `u32`, `i32`. Avoid `usize`, `u64`, `isize` in `#[cube]` kernels.

**Rule 5 — `erf` availability:**
`erf` is a C standard math function. CubeCL 0.9.x CPU backend exposes it as `f64::erf()` via the Erf trait. For wgpu, `erf` is not a native WGSL builtin — requires polynomial approximation (Horner scheme, e.g., Abramowitz & Stegun 7.1.26 for 1e-7 accuracy). This is a **CRITICAL FINDING** — see Common Pitfalls.

### Pattern 3: CubeCL `#[cube]` Struct for PairData (D-15)

CubeCL 0.9.x supports structs inside `#[cube]` when annotated with `#[derive(CubeType)]`:

```rust
use cubecl::prelude::*;

#[derive(CubeType, Clone, Copy)]
pub struct PairData {
    pub zeta_ab: f64,
    pub center_p: [f64; 3],   // NOTE: fixed-size array in CubeType needs verification
    pub rirj: [f64; 3],
    pub fac: f64,
    pub aij2: f64,
}
```

**Risk:** Fixed-size `[f64; 3]` arrays inside a `CubeType` struct may not be supported in CubeCL 0.9.x. If compilation fails, fall back to passing individual fields as separate `f64` parameters (6 scalars for 2D, or use separate x/y/z parameters).

The `#[cube]` struct fallback pattern:
```rust
// If [f64; 3] in CubeType fails, pass components as scalars:
#[cube]
pub fn compute_pdata(
    ai: f64, aj: f64,
    rix: f64, riy: f64, riz: f64,
    rjx: f64, rjy: f64, rjz: f64,
    norm_i: f64, norm_j: f64,
    // outputs written to array slices
    pdata_buf: &mut Array<f64>,  // layout: [zeta, px, py, pz, aij2, fac, ...]
) { ... }
```

### Anti-Patterns to Avoid
- **Calling `gamma_inc_like` from the C oracle inside `#[cube]` code:** The oracle is host-only. Math `#[cube]` functions must be self-contained.
- **Using recursion in `#[cube]` functions:** CubeCL does not support recursive `#[cube]` calls. All loops must be iterative with bounded `while`/`for` using `u32` counters.
- **Modifying global mutable state in `#[cube]`:** All outputs must flow through `&mut Array<f64>` parameters or return values.
- **Using `Vec<f64>` inside `#[cube]`:** Use `Array<f64>` (CubeCL's device array) or local register variables only.

| Problem | Don't Build | Use Instead | Why |

**Key insight:** libcint's vendored source is the only required "don't hand-roll" target. Every algorithm already exists in `libcint-master/src/`. The work is porting to CubeCL, not inventing math.

## Common Pitfalls

### Pitfall 1: `erf()` Not Available in wgpu Backend
**What goes wrong:** `f64::erf(t)` in a `#[cube]` function compiles on CPU backend but fails or silently produces wrong values on wgpu because WGSL does not have a native `erf` instruction.
**Why it happens:** CubeCL maps `f64::erf` to the backend's math library; on CPU this is glibc `erf()`, but on wgpu it must be emulated.
**How to avoid:** For Phase 8, CPU backend is the primary validation target (D-18, D-22). For wgpu validation (D-18 "opportunistically"), either (a) implement a high-accuracy polynomial approximation of `erf` as a `#[cube]` function, or (b) restrict wgpu Boys validation to the power series branch (large x: asymptotic via erfc is optional on wgpu until confirmed available). Document which branch is tested on each backend.
**Warning signs:** Boys values on wgpu diverge from CPU for any `t >= TURNOVER_POINT[m]`.

### Pitfall 2: E0433 — Calling Plain Rust from `#[cube]`
**What goes wrong:** A non-`#[cube]` helper called from inside a `#[cube]` function produces `"function X is not a crate or module (E0433)"`.
**Why it happens:** CubeCL's `#[cube]` macro transforms function call resolution into IR expansion. Plain Rust functions are not expandable.
**How to avoid:** Every function called from `#[cube]` must itself be `#[cube]`. Annotate ALL helpers (`double_factorial`, `convergence_check`, etc.) with `#[cube]`.
**Warning signs:** Any E0433 error mentioning a function name that appears inside a `#[cube]` block.

### Pitfall 3: if-expression type mismatch (E0308 `ExpandElementTyped`)
**What goes wrong:** `let val = if cond { expr1 } else { expr2 };` inside `#[cube]` causes `mismatched types: expected ExpandElementTyped<_> found {float}`.
**Why it happens:** CubeCL wraps IR values in `ExpandElementTyped<T>`; if-expressions don't unify the branch types correctly.
**How to avoid:** Always use the statement form: `let mut val: f64 = default; if cond { val = expr1; } else { val = expr2; }`. The Boys function has an if/else for t==0 / t<turnover / t>=turnover — all three branches must be written as statement-form.
**Warning signs:** E0308 with "ExpandElementTyped" in the error message.

### Pitfall 4: Wheeler Fallback Requires `u64` Math
**What goes wrong:** Wheeler's algorithm in `rys_wheeler.c` uses double-precision iteration; Rust translation may use `usize` for loop bounds, which is unsupported in `#[cube]`.
**Why it happens:** Natural Rust idioms use `usize` for array indices.
**How to avoid:** Use `u32` for all loop counters and array indices inside `#[cube]`. Cast to `u32` at entry boundaries.
**Warning signs:** Compiler error: "usize is not CubeType".

### Pitfall 5: `[f64; 3]` in `CubeType` structs may fail
**What goes wrong:** `#[derive(CubeType)]` on a struct containing `[f64; 3]` may not expand correctly in CubeCL 0.9.x.
**Why it happens:** CubeCL's `CubeType` derive handles scalars well but fixed-size arrays may require special treatment (e.g., using `cubecl::Array<f64>` instead of `[f64; 3]`).
**How to avoid:** During Wave 0, write a minimal compile test of `PairData` as `#[derive(CubeType)]` with `[f64; 3]` fields. If it fails, switch to separate x/y/z scalar fields (`center_p_x: f64, center_p_y: f64, center_p_z: f64`).
**Warning signs:** CubeCL macro expansion errors mentioning the struct's array field.

### Pitfall 6: TURNOVER_POINT array indexing with dynamic `m`
**What goes wrong:** `TURNOVER_POINT[m as usize]` inside `#[cube]` fails because `usize` cast or const-array indexing with `u32` may not be supported.
**Why it happens:** CubeCL `#[cube]` may not allow dynamic indexing of `const` arrays in all backends.
**How to avoid:** Use a `match m { 0 => 0.0, 1 => 0.0, 2 => 0.866..., ... }` pattern or restructure the Boys function to receive `turnover` as a pre-computed parameter from host-side dispatch. Alternatively, pass `turnover` as a `f64` parameter to `boys_fm()`. Verify const-array indexing works with a minimal test in Wave 0.
**Warning signs:** Compiler error on the `TURNOVER_POINT[m]` access inside `#[cube]`.

### Pitfall 7: Oracle harness build gate
**What goes wrong:** Oracle calls to `gamma_inc_like` only compile when `CINTX_ORACLE_BUILD_VENDOR` is set (build.rs gate). Math validation tests that call oracle without this env var silently get no reference values.
**Why it happens:** The oracle build is gated to keep CI fast (see `cintx-oracle/build.rs`).
**How to avoid:** In math validation tests, either (a) set `CINTX_ORACLE_BUILD_VENDOR=1` in the test environment, or (b) use pre-computed golden values for the initial test pass and add oracle-live comparison as a separate test target. Document which tests require the oracle build.
**Warning signs:** Tests pass trivially because oracle values return 0 or panic silently.

## Code Examples

### Boys function — iterative power series (small x branch)
```rust
// Source: libcint-master/src/fmt.c, fmt1_gamma_inc_like()
// Translated to CubeCL-compatible Rust

#[cube]
fn boys_power_series(t: f64, m: u32, f: &mut Array<f64>) {
    // b = m + 0.5
    let b = m as f64 + 0.5;
    let e = 0.5 * f64::exp(-t);
    let mut x = e;
    let mut s = e;
    let tol = DBL_EPSILON_HALF * e;  // const f64 = f64::EPSILON * 0.5

    // Converging series: x_{k+1} = x_k * t / (b+k+1)
    let mut bi = b + 1.0;
    while x > tol {
        x = x * t / bi;
        s = s + x;
        bi = bi + 1.0;
    }
    // f[m] = s / b, then downward recurrence
    let mut val = s / b;
    f[m] = val;
    let mut i = m;
    let mut b_r = b;
    while i > 0 {
        b_r = b_r - 1.0;
        val = (e + t * val) / b_r;
        f[i - 1] = val;
        i = i - 1;
    }
}
```

### Boys function — erfc + upward recurrence (large x branch)
```rust
// Source: libcint-master/src/fmt.c, gamma_inc_like() large-x branch
#[cube]
fn boys_erfc_recurrence(t: f64, m: u32, f: &mut Array<f64>) {
    let tt = f64::sqrt(t);
    // F_0(t) = sqrt(pi/4) / sqrt(t) * erf(sqrt(t))
    // NOTE: f64::erf() requires Erf trait from cubecl; wgpu needs approx
    f[0] = SQRTPIE4 / tt * f64::erf(tt);
    let e = f64::exp(-t);
    let b = 0.5 / t;
    let mut i: u32 = 1;
    while i <= m {
        f[i] = b * ((2 * i - 1) as f64 * f[i - 1] - e);
        i = i + 1;
    }
}
```

### Obara-Saika overlap VRR (1e, no Rys roots needed)
```rust
// Source: libcint-master/src/g1e.c, CINTg1e_ovlp() lines 125-183
#[cube]
fn vrr_overlap_1d(
    g: &mut Array<f64>,    // g[0..nmax+1] for one Cartesian dimension
    rijrx: f64,            // rij[d] - rx[d]
    aij2: f64,             // 0.5 / (ai + aj)
    nmax: u32,
    di: u32,               // stride along i-index
) {
    // g[di] = rijrx * g[0]
    g[di] = rijrx * g[0];
    let mut i: u32 = 1;
    while i < nmax {
        g[(i + 1) * di] = i as f64 * aij2 * g[(i - 1) * di] + rijrx * g[i * di];
        i = i + 1;
    }
}
```

## Open Questions

1. **`f64::erf()` in CubeCL wgpu backend**
   - What we know: CPU backend exposes `erf` via the Erf trait. WGSL has no native erf.
   - What's unclear: Does CubeCL 0.9.x emulate erf in WGSL, or does it error?
   - Recommendation: During Wave 0, write a one-liner `#[cube]` function that calls `f64::erf(0.5)` and test on wgpu. If it fails, implement a polynomial approximation and gate it behind a `#[cfg]` check.

2. **Const array indexing with dynamic u32 in `#[cube]`**
   - What we know: CubeCL embeds const values as literals in the IR. Dynamic indexing of a const array inside `#[cube]` may require the array to be "unrolled" or the index to be a compile-time constant.
   - What's unclear: Whether `TURNOVER_POINT[m]` with runtime `m: u32` compiles to valid IR for the CPU backend.
   - Recommendation: Wave 0 minimal test: `#[cube] fn test_idx(m: u32) -> f64 { TURNOVER_POINT[m] }`. If it fails, refactor to a `match` expression or pass `turnover` as a parameter.

3. **`#[derive(CubeType)]` with `[f64; 3]` fields**
   - What we know: CubeCL 0.9.x supports CubeType structs with scalar fields.
   - What's unclear: Whether fixed-size array fields (`[f64; 3]`) are supported.
   - Recommendation: Wave 0 minimal test. If it fails, use separate `_x/_y/_z` scalar fields per D-15 discretion.

4. **Oracle build for math-specific functions**
   - What we know: `cintx-oracle` currently builds compat/raw API fixtures, not raw `gamma_inc_like`/`CINTrys_roots` calls.
   - What's unclear: Whether the oracle harness needs a new `math_compare.rs` module with FFI bindings to the internal libcint functions (not exposed through the public C API).
   - Recommendation: Add `extern "C" { fn gamma_inc_like(...); fn CINTrys_roots(...); }` bindings gated under `cfg(oracle-vendor)` to `cintx-oracle/src/math_oracle.rs`. These internal functions are defined in `libcint-master/src/fmt.c` and `rys_roots.c` and are compiled when the cc build runs.
