# Phase 8: Gaussian Primitive Infrastructure and Boys Function - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 8 implements all shared mathematical primitives that integral kernels will consume: Boys function, Gaussian pair data (pdata), Rys quadrature roots/weights, and Obara-Saika horizontal/vertical recurrence relations. All are implemented as validated `#[cube]` functions, confirmed against libcint reference values before any kernel consumes them. This is pure math infrastructure — no kernel integration, no API changes.

</domain>

<decisions>
## Implementation Decisions

### Boys Function
- **D-01:** Replicate libcint's `fmt.c` table-based approach. Upload gridded Taylor expansion coefficients as compile-time constants in `#[cube]` code (const arrays, zero upload overhead).
- **D-02:** Match libcint's compile-time MMAX for maximum Boys order m. Whatever `fmt.c` defines as its maximum supported order.
- **D-03:** Replicate libcint's erfc-based asymptotic formula for the large-x branch. Same mathematical approach as `fmt1_erfc_like` in `fmt.c`.
- **D-04:** Turn-over threshold between small-x (table) and large-x (asymptotic) branches may be optimized for GPU (reduced branch divergence), as long as output values match libcint reference within 1e-12 atol.
- **D-05:** Grid spacing for the Taylor table may be tuned for GPU cache efficiency, as long as accuracy is maintained within tolerance.
- **D-06:** Taylor term count per grid interval may differ from libcint if accuracy within 1e-12 is maintained. Allows balancing accuracy vs GPU register pressure.
- **D-07:** Single `boys_fm(x, m)` `#[cube]` function with internal if/else branching (not separate per-region functions). Simpler call site at kernel level.
- **D-08:** Domain coverage for validation: match libcint exactly — same grid points, same reference values. Byte-for-byte compatible reference data.

### Rys Quadrature
- **D-09:** Polynomial fit coefficients from `polyfits.c` provided as compile-time constants in `#[cube]` code (matching Boys function approach — D-01).
- **D-10:** Support all quadrature degrees that libcint's `polyfits.c` supports. Full coverage of the upstream maximum.
- **D-11:** Polynomial fits for common degrees, Wheeler algorithm fallback for edge cases beyond fit coverage. Both implemented as `#[cube]` functions.

### #[cube] Function Design Patterns
- **D-12:** Shared math `#[cube]` functions live in a new `crates/cintx-cubecl/src/math/` module with files: `boys.rs`, `rys.rs`, `obara_saika.rs`, `pdata.rs`. Clean separation from kernel dispatch.
- **D-13:** Compile-time parameters (max angular momentum, table sizes) use fixed `const` values (hardcoded), not const generics. Simpler #[cube] compatibility.
- **D-14:** Obara-Saika horizontal and vertical recurrence implemented as separate `#[cube]` functions: `hrr_step()` and `vrr_step()`. Kernel families compose them as needed.
- **D-15:** Gaussian pair data (pdata) uses a `#[cube]` struct with named fields (`zeta_ab`, `center_p`, etc.), not raw f64 slices. CubeCL struct support must be verified during research.

### Validation and Oracle Comparison
- **D-16:** Reference values generated by oracle harness calling libcint C functions directly (`fmt.c`, `polyfits.c`, etc.). Most authoritative source.
- **D-17:** Per-function unit test suites (`boys_tests.rs`, `rys_tests.rs`, etc.) plus a combined integration test that chains math primitives. Both unit and integration validation.
- **D-18:** CPU backend is the primary validation target (--features cpu, f64 guaranteed). If wgpu with SHADER_F64 is available, also run on wgpu to catch backend-specific issues early.
- **D-19:** Tolerance strategy: absolute-only, 1e-12 atol (`|computed - reference| < 1e-12`). No relative error checking.

### Carried Forward from Phase 7
- **D-20:** Buffer lifecycle lives inside kernel family modules, not centralized (Phase 7 D-07). Math functions are pure computation, no buffer management.
- **D-21:** Both backends produce f64 results; wgpu gates on SHADER_F64 (Phase 7 D-09).
- **D-22:** CPU backend is the oracle-parity target (Phase 7 D-03).

### Claude's Discretion
- Exact const array layout for Taylor coefficients and polyfit tables
- Internal helper functions for erfc computation within #[cube]
- Wheeler algorithm implementation details for Rys fallback
- Exact field names and layout of the PairData #[cube] struct
- Test fixture design for reference value generation
- How to handle CubeCL #[cube] struct limitations if they arise (fallback to named-field modules)

</decisions>

## Existing Code Insights

### Reusable Assets
- `crates/cintx-oracle/` — Oracle harness for calling libcint C functions; can generate reference values for Boys, Rys, pdata
- `crates/cintx-cubecl/src/backend/mod.rs` — ResolvedBackend with Cpu arm for validation; math #[cube] functions will be tested against this
- `libcint-master/src/` — Complete vendored libcint source: fmt.c, polyfits.c, rys_roots.c, g1e.c, g2e.c are the oracle truth

### Established Patterns
- `#[cube]` functions in CubeCL: see `docs/cubecl_error_guideline.md` for restrictions
- Kernel family stubs accept `&ResolvedBackend` + `&mut [f64]` staging — math functions will be called from within these stubs
- Test pattern: `cargo test -p cintx-cubecl --features cpu` for CPU-backend validation

### Integration Points
- Math functions in `src/math/` will be imported by kernel stubs in `src/kernels/`
- Boys function is consumed by all kernel families (1e, 2e, 2c2e, 3c1e, 3c2e, 4c1e)
- Rys quadrature consumed by 2e-type families (two_electron, center_2c2e, center_3c2e)
- Obara-Saika recurrence consumed by all families for auxiliary integral computation

</code_context>

<specifics>
## Specific Ideas

- Boys function table: replicate fmt.c's mathematical approach faithfully, but allow GPU-friendly tuning of grid spacing and turn-over thresholds
- Rys: polynomial fits for common degrees + Wheeler fallback for edge cases — both as #[cube]
- All math validation runs against libcint C oracle calls, not precomputed golden files
- PairData as a typed #[cube] struct, not raw arrays — verify CubeCL struct support during research

</specifics>
